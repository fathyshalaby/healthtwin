# What's Missing

> Honest gap inventory for HealthTwin. Phases 1–5 are built and tested, but "tested against fakes" ≠ "production-ready." Generated by an independent 6-agent audit (50 raw findings, de-duplicated below).

## Recommended next 3 moves

1. **Make the packages actually installable + built.** The product IS an npm SDK, and today no package is consumable off-workspace (every `packages/*/package.json` points `main`/`types` at raw `src/index.ts`, no bundler, no `exports`, all `0.0.0`). Add a real build (tsup/rollup → `dist/` ESM+CJS+dts), fix manifests, and publish something. Nothing else matters if partners can't `npm install`.
2. **Stand up CI + verify the RLS boundary.** There is no `.github/` at all and the only live test signs in as one user round-tripping its own row — the sole authorization control for all PHI is unverified. Add a GitHub Action running `pnpm install --frozen-lockfile` + `turbo run typecheck test build` against a local Supabase, and write the cross-user-denial + consent expiry/revoke tests the spec mandates.
3. **Build the missing write/share/back-of-body UI paths.** Three designed-but-half-wired core features are non-functional: edit/delete (immutable-log write half never built), the anterior/posterior toggle (every surface hardcodes `anterior`, so lower-back is uncapturable), and consent grants (full SQL, zero client surface). These are small-to-medium UI wiring on top of plumbing that already exists.

## 1. Blocking a real MVP

**Packages are not installable or built as an SDK.** The whole product is an npm SDK, but every package points `main`/`types` at uncompiled `src/index.ts`, ships only loose per-file ESM via bare `tsc` (no CJS, no bundled dts, leaked `*.test.js` in `dist/`), has no `exports`/`files`/`publishConfig`, and is pinned `0.0.0` — a partner literally cannot `npm install @healthtwin/react`. **Effort: M.** First step: add tsup/rollup producing `dist/index.js` (ESM+CJS) + `dist/index.d.ts`, rewrite manifests (`main`/`module`/`types`/`exports`/`files`/`prepublishOnly`).

**Framework-agnostic embed has no bundle.** `embed.html` loads `./embed.bundle.js` which is never produced (no bundler dep anywhere); `tsc` emits bare `import ... from "@healthtwin/react"` specifiers a browser can't resolve, so the headline "no React required" iframe/web-component path is non-functional. **Effort: M.** First step: add a tsup/vite build that inlines React/ReactDOM into a single self-contained `embed.bundle.js`.

**Partner token-exchange server + `partner_id` tenant scoping don't exist.** Only the client helper (`packages/embed/src/tokenExchange.ts`) exists — no server verifies a partner token or mints a scoped session, and there is no `partner_id` column in any migration/`ObservationRow`/core type (RLS is only `subject_id = auth.uid()`), so all anonymous partner users collide on `subjectId='embed-anon'` with no tenant isolation. This is the load-bearing multi-tenant trust boundary and it's absent. **Effort: L.** First step: add a `partner_id` column + RLS scoping in a migration, then a Next.js route that verifies the partner-signed JWT and mints a scoped Supabase session.

**RLS / consent / audit are entirely unverified.** RLS is the sole authorization mechanism for all PHI, yet the only live test (`packages/supabase/src/live.test.ts`) is `skipIf(!live)`, signs in as one user, and asserts a happy-path round-trip — no cross-user-denial test, no consent grant/expiry/revoke test, no audit-row assertion, and there is no CI to run it. A broken policy would pass every current test. **Effort: M.** First step: add a local-Supabase harness with two seeded users and a test asserting user B's read of user A's rows is denied.

**No CI at all.** There is no `.github/workflows`; the 49 unit + 2 e2e tests, axe assertions, and RLS tests only ever run via a manual `pnpm -w test`, so regressions ship silently in a partner-facing SDK. **Effort: S.** First step: add one GitHub Action running `pnpm install --frozen-lockfile` + `turbo run typecheck test build` + Playwright.

**Edit/delete an entry is impossible.** The core defines an immutable log with `supersedes`/`tombstone` and `foldLog` reads them, but the write half was never built — `NewObservation`/`createObservation` omit them, the provider exposes only `add`, and `EntriesList`/`Timeline` render plain `<li>` with no edit/delete affordance, so users can't correct a mistap and GDPR erasure has no client trigger. **Effort: M.** First step: add `supersedes`/`tombstone` to `createObservation` + an `edit`/`remove` on `useObservations`, then wire a control in `EntriesList`.

**Back-of-body capture is impossible.** Every surface hardcodes `view="anterior"` (`apps/web/app/page.tsx:12`, `review/page.tsx:11`, `apps/native/.../CaptureScreen.tsx:17`, `embed.html:13`) and there's no view toggle anywhere; since `lower_back` is `views:['posterior']` only (`taxonomy.ts:11`), the single most common MSK complaint can't be logged. **Effort: S.** First step: add an anterior/posterior toggle wired to `BodyMapCapture`'s `view` prop.

**Consent/sharing has no client surface.** The `consent_grants` table with scoped, time-boxed, revocable grants + RLS is fully built (`0002_consent_grants.sql`), but the supabase package exports only `adapter/auth/rowMapping` — no create/list/revoke grant function, hook, or UI — so the product's headline "share with your clinician" is entirely non-functional. **Effort: M.** First step: add `createGrant`/`listGrants`/`revokeGrant` to `packages/supabase/src/index.ts` and a minimal share/revoke UI.

**Review has no filtering or region→timeline linkage.** `BodyMapReview.tsx:36` passes `onSelect={() => {}}` (tapping a region does nothing), there's no date-window control feeding `buildTimeline`/`computeHeatmap` (which accept `from`/`to`), and `Timeline` renders all history unfiltered — so the heatmap aggregates all-time forever, defeating the longitudinal "what happened this week vs last month" differentiator. The query layer (`queryObservations`/`ObservationFilter`) already supports it; nothing wires it. **Effort: M.** First step: wire `onSelect` to a region filter and add a date-range selector supplying `from`/`to`.

**Native app has zero runtime tests.** The four real RN files (`App.tsx`, `nativeProvider.tsx`, `CaptureScreen.tsx`, `BodyMapNative.tsx`) have no tests, no RN Testing Library, no `test` script, and `apps/native` is excluded from the workspace (`!apps/native`), so `turbo run test` never touches a whole shippable capture→save→list surface the spec commits to test. **Effort: M.** First step: add RN Testing Library + a `test` script and cover `CaptureScreen`'s save flow.

## 2. Needed before real users/partners (production hardening)

**GDPR/CCPA erasure + export are unbuilt and architecturally conflicted.** No purge job/function, no export function, and the observations table is intentionally append-only with no delete policy — so a tombstone can never actually remove plaintext, which the research brief says does *not* satisfy Art. 17; there's also no user-facing export or delete-account action anywhere. **Effort: L.** First step: add a hard-purge SQL function (incl. backup strategy) plus an export endpoint, then surface both in-app.

**Column-level note encryption is documented but absent.** The Art. 9 free-text `note` is stored plaintext (`0001` carries a literal `TODO: column-encrypt via pgcrypto`), yet the README/spec advertise column-level encryption — marketing a control that doesn't exist is exactly the GoodRx/BetterHelp FTC trap. **Effort: M.** First step: encrypt `note` via pgcrypto in the adapter write path + a migration.

**Embed broadcasts PHI to `postMessage(..., '*')`.** `embedElement.tsx:36` and `embed.html:20` post captured health observations to any parent origin with no allowlist and no receiver-side origin check — a textbook insecure-postMessage disclosure for a cross-origin widget. **Effort: S.** First step: add a required `allowed-origin` attribute and pass it as the `targetOrigin`.

**No sign-out / session management.** `signOut` exists in `auth.ts` but is never wired; `apps/web/app/providers.tsx` reads `currentUserId` once, never subscribes to `onAuthStateChange`, never refreshes, and stores the session in localStorage with no SSR/PKCE — an un-endable session on a shared device is real exposure for a health app. **Effort: M.** First step: wire `signOut` to a UI control and add `onAuthStateChange`.

**No onboarding / consent-to-capture / privacy notice.** Cloud mode drops users straight into a bare email-OTP form with no disclosure of what health data is collected — but Art. 9 special-category data legally requires explicit, granular, informed consent *before* capture. **Effort: M.** First step: add a consent-to-collect screen gating first capture.

**Consent-grant policies have concrete bugs.** Beyond the missing app layer, `grants_update_own` has a `USING` but no `WITH CHECK` (a grantor can mutate grantee/scope/grantor post-hoc), and `observations_select_granted` has no tombstone filter (deleted observations keep flowing to clinicians) — undermining the revocability the product promises. **Effort: M.** First step: add `WITH CHECK` to the update policy and a tombstone filter to the granted-select policy.

**Audit log covers only inserts.** The `0003` trigger fires on observation INSERT only — no read/share/export auditing, and `audit_select_own` scopes to `actor = auth.uid()`, so a subject can't see *which clinicians read their data*, gutting the doctor-portal accountability story. **Effort: M.** First step: add grant/revoke/read audit events and let subjects see accesses to their own data.

**Sync has no error/offline/retry handling or tests.** The provider calls `await runSync(...)` with no `.catch` (an unhandled rejection on failure), there's no backoff/retry despite the spec's "syncs reliably… retries with backoff" promise, and `sync.test.ts` covers only happy paths — the most operationally important behavior of an offline-first engine is untested. **Effort: M.** First step: add retry/backoff + error surfacing in the sync effect and a failing-adapter test.

**No rate limiting, security headers, or middleware.** No CSP/X-Frame-Options/CORS/CSRF/rate-limit anywhere; writes go browser→Supabase with only the anon key, so a scripted client can flood observations or email-bomb the OTP endpoint, and missing headers leave the app open to clickjacking. **Effort: M.** First step: add security headers in `next.config.mjs` and throttle OTP/insert volume.

**Native app is far below web parity.** `CaptureScreen.tsx` captures only intensity with `type` hardcoded to `'pain'` (no type/quality/note/tags/time/point), has no review/heatmap/timeline, no posterior view, and its provider is local-only with no `SyncAdapter` — so a mobile user's twin never reaches the cloud or a clinician. **Effort: M.** First step: wire the `SyncAdapter` into `nativeProvider` and add a type/quality picker.

**No column tests on the web app source.** `providers.tsx` encodes the security-critical rule that in cloud mode `subjectId` must equal the authenticated uid (`:44`) plus the sign-in gate, and `getCloudConfig`'s env-gated local-vs-cloud branch — all untested; the two Playwright specs cover only the local happy path. **Effort: M.** First step: add a vitest unit test pinning the `subjectId === uid` invariant.

**`live.test.ts` never runs; no local-Supabase harness.** `skipIf(!live)` + no CI means the adapter's real behavior against Postgres+RLS is never exercised on any commit — effectively dead code. **Effort: M.** First step: add `supabase start` + a seeded confirmed user to CI so the integration test actually runs. *(Closes with the CI + RLS items above.)*

**No theming/branding API for partners.** Components take no `className`/`style`/`theme`, there are zero CSS variables, and the embed renders into light DOM with no Shadow DOM or `observedAttributes` — so a partner's embedded widget can't match their brand and looks foreign, contradicting the spec's first-class theming criterion. **Effort: M.** First step: expose design-token CSS variables and a `className` passthrough; move the embed into Shadow DOM.

**No stable/versioned public API.** Every `index.ts` does `export *` including internals (`idbStore`, `idbSyncMeta`, store/sync/log guts), with no `@public`/`@internal` split, api-extractor, or semver policy — any refactor is a silent breaking change for partners. **Effort: M.** First step: curate explicit public exports and add api-extractor with an API report.

**No accessibility coverage beyond the body map.** Only `bodymap-react` runs axe; `EntrySheet` (intensity slider, labels, focus-on-save), `BodyMapCapture`, review surfaces, and the embed have zero a11y assertions despite the spec calling a11y "first-class." **Effort: M.** First step: add an axe test for `EntrySheet`.

**Under-tested embed/token wire contracts.** `embedElement.test.tsx` asserts only the CustomEvent path, not the `postMessage({type:'healthtwin:observation', observation})` shape partners integrate against; `exchangePartnerToken` has no test for fetch-throws, malformed JSON, or a 200 missing `token`/`subjectId`. **Effort: S.** First step: add a postMessage-shape contract test + tokenExchange error-path tests.

**`expoSqliteAdapter` (real device binding) is untested.** The only code talking to the real driver branches on params-length to pick `runAsync` vs `execAsync` (`:19-25`) and is bypassed by `fakeSqlDb`; an empty-params query silently routing wrong would never be caught. **Effort: S.** First step: add a fake `ExpoSqliteDatabase` recording routing and pin the branch.

**No operational compliance scaffolding.** No BAA/DPA templates or sub-processor list, no EU-residency config (US-hosted Supabase/Vercel triggers GDPR Chapter V + CLOUD Act exposure), no DPIA/RoPA, no secrets store for the future service-role key, no breach runbook. **Effort: L.** First step: document residency + sub-processors and add DPA/BAA templates.

**No release/versioning pipeline.** All `0.0.0`, no changesets/CHANGELOG/release scripts/tags, and `workspace:*` deps must be rewritten to real ranges at publish — there's no tool to do it. **Effort: M.** First step: add changesets and a publish workflow that rewrites workspace deps.

**No deploy config for the hosted backend.** `apps/web` has no `vercel.json`/Dockerfile/`fly.toml`, and `next.config.mjs` sets no `output` mode — the "hosted backend" half can't be reproducibly deployed. **Effort: M.** First step: pick a target and commit either `output:'standalone'`+Dockerfile or `vercel.json` with env wiring.

**No observability or error tracking.** No Sentry, no `instrumentation.ts`, no structured logging; sync/token-exchange failures are silent to operators, so PHI-handling incidents are invisible. **Effort: S.** First step: add an error client + `instrumentation.ts` and surface sync errors.

**No runtime/toolchain pinning.** No `.nvmrc`/`.node-version`/`engines` despite the plan's "Node 24 LTS," pnpm drifted `9.12.0`→`10.23.0` undocumented, and all deps are caret-ranged — CI/contributor envs can silently diverge. **Effort: S.** First step: add `engines`+`.nvmrc` and enforce `--frozen-lockfile`.

**No developer docs / API reference / per-package READMEs.** `docs/` has 3 internal files; only 3 of 7 packages have a README; no getting-started, no typedoc, no reference site. **Effort: L.** First step: add typedoc + a getting-started guide covering hooks/components/adapter contracts.

**No partner example/starter.** Nothing demonstrates the *external* integration path (npm-installed SDK, `exchangePartnerToken` against a mock partner server, bring-your-own-backend adapter, CDN embed) — each partner must reverse-engineer the flow. **Effort: M.** First step: ship a minimal partner app doing token handoff + a custom `LocalStore` adapter.

## 3. Nice-to-have / later

**Higher-order product surfaces (AI insights, doctor portal, weekly review, churn/injury dashboards).** Correctly deferred (spec YAGNI) and out of capture-engine v1 scope, but this is where the research brief locates the actual defensible wedge and pricing power — plus the SaMD/AI-Act regulatory fork to design for before shipping. **Effort: L.** First step: scope a doctor-portal/weekly-review MVP as the next phase.

**Internationalization.** No i18n library and all strings/labels are hardcoded English literals (raw slugs like `'pain'`, English-only taxonomy labels, `YYYY-MM-DD` dates) — a structural gap for the India-first/EU partner-SDK go-to-market that's cheaper to design in now. **Effort: M.** First step: externalize strings behind an i18n hook and add locale date formatting.

**No visual/screenshot regression tests.** For an SVG body-map product, geometry regressions in `bodymap-core` (`shapesFor`/path data) that misalign or break a region pass all tests as long as aria-labels resolve. **Effort: M.** First step: add a Playwright `toHaveScreenshot` baseline for anterior/posterior maps.

**No load/perf tests or coverage measurement.** The immutable log is fully re-folded via `useMemo(() => foldLog(all))` on every render and grows unbounded with no pagination on pull — a latent perf cliff — and no `--coverage`/thresholds means untested surfaces aren't even visible as numbers. **Effort: M.** First step: add a `foldLog`/`runSync` benchmark over thousands of observations and turn on coverage.

**Broaden module-format / React compatibility.** ESM-only with React pinned `^19.0.0` excludes the large share of 2026 partner apps still on React 18 and any CJS toolchain. **Effort: M.** First step: widen the peer range to `^18 || ^19` and ship CJS (rides on the build/publish work). *(Prereq: the build machinery in §1.)*

**No lint/format enforcement.** No ESLint/Prettier config or scripts, so common bugs (`no-floating-promises`, react-hooks rules, unused vars) go uncaught beyond `tsc` strict in an open-PR repo. **Effort: S.** First step: add typescript-eslint + react-hooks + prettier with a `lint` script in CI.

**No dependency-update automation.** No Renovate/Dependabot plus caret ranges and no CI means CVEs in `next`/`react`/`@supabase/supabase-js`/`expo` drift unnoticed — real supply-chain exposure for a health SDK. **Effort: S.** First step: add `renovate.json` (or `dependabot.yml`).

**Missing repo governance docs.** LICENSE exists, but no `SECURITY.md` (no disclosure channel despite handling health data), `CONTRIBUTING.md`, `CODEOWNERS`, or `CODE_OF_CONDUCT.md` for a repo inviting external PRs. **Effort: S.** First step: add `SECURITY.md` with a disclosure address, then CONTRIBUTING/CODEOWNERS.
